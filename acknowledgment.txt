================================================================================
  PLATFORM OVERVIEW
================================================================================

Savion is a full-stack, production-grade PG (Paying Guest) accommodation 
management platform built with Next.js 15, Firebase, and Framer Motion. 
It serves four distinct user roles with role-based dashboards, real-time 
data sync, and automated lifecycle management.

================================================================================
  USER ROLES & CAPABILITIES
================================================================================

  [ADMIN]
  • Full platform control center with animated tab navigation
  • Real-time user, property, and booking oversight
  • Verification document review and approval workflow
  • Identity (KYC / Aadhaar) approval queue

  [OWNER]
  • Premium dashboard with live revenue trajectory chart (5 time filters)
  • Platform Pulse — real-time activity feed (payments, complaints, bookings)
  • Bento-grid property portfolio
  • UPI payment gateway integration
  • Booking inquiry approval flow

  [CARETAKER]
  • Property management with real-time status sync
  • Complaint monitoring and resolution tracking

  [TENANT]
  • Smart booking wizard with KYC identity verification bypass for verified users
  • Digital rental agreement with e-signature
  • Payment hub with UPI QR-code flow
  • Move-out notice system (7-day checkout timeline)
  • Review and rating system (post-stay)
  • Complaint filing system

================================================================================
  BOOKING LIFECYCLE (ENFORCED)
================================================================================

  pending → confirmed → notice_given → notice_approved → [AUTO CHECKOUT @ 5 PM]

  On Auto-Checkout (Hard Cleanup):
  ✓ Room availability RESTORED (occupancy-- and available++)
  ✓ All complaints for that tenant-PG pair deleted
  ✓ Contract record deleted from Firestore
  ✓ Booking record deleted from Firestore (last, to prevent re-fire)
  ✓ Payment history PRESERVED
  ✓ User reviews PRESERVED

  Re-Booking Gate Logic:
  ✗ Cannot book if status is: confirmed, notice_given, notice_approved
  ✓ Can book if: no active booking, or all prior records deleted post-checkout

================================================================================
  BUGS FOUND & FIXED (FULL AUDIT)
================================================================================

  --- Session 1 (Core Logic) ---
  1. JSX tag mismatch in tenant/page.tsx (</div> → </motion.div>)
  2. Map function closing tag mismatch in tenant/page.tsx
  3. Booking guard only blocked confirmed+pending — now also blocks
     notice_given and notice_approved statuses
  4. Auto-vanish only mutated status; now performs hard Firestore deletes
  5. Revenue chart used hardcoded mock data; now reads live rawPayments
  6. Activity feed used mock items; now reads real Firestore events
  7. "View All Activity" button had no onClick handler; now toggles feed
  8. Header "Book Now" button had no active-booking check; now disabled
     with contextual reason text

  --- Session 2 (Deep Audit — Firebase Permissions + Data Integrity) ---
  9.  🔴 CRITICAL: Complaint interface missing `ownerId` field.
      Owner dashboard queried `where("ownerId", "==", ownerId)` on complaints
      collection but complaints never stored ownerId.
      → Caused: "Missing or insufficient permissions" Firebase error
      → Fixed: Added `ownerId` to Complaint interface + createComplaint call

  10. 🟡 MEDIUM: `restoreRoomAvailability()` used `r.id === roomId` but
      PGRoom interface only has `roomNumber`, not `id`. Room matching
      would NEVER succeed, making rooms appear permanently occupied.
      → Fixed: Now matches on `r.roomNumber === roomId || (r as any).id === roomId`

  11. 🟡 MEDIUM: Owner dashboard `timeAgo()` typed parameter as `string`
      but `createdAt` is stored as `number` (Unix milliseconds). This caused
      incorrect or NaN time calculations in the activity feed.
      → Fixed: Accepts `string | number`, handles both via typeof check

  12. 🟡 MEDIUM: Auto-Vanish effect re-triggered infinitely. useEffect
      depended on `[bookings, contracts]` — deleting a booking fires
      onSnapshot, updates bookings state, re-triggers effect, tries to
      delete again (fails silently but wastes resources/logs errors).
      → Fixed: Added `deletedBookingsRef` (useRef<Set>) to skip already-
        processed booking IDs. On error, ID is removed so retry works.
